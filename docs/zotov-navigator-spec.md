# Зотов.Навигатор — Техническая документация проекта

> Веб-приложение для digital-навигации внутри Центра «Зотов» (бывший Хлебозавод №5).
> Хакатон «Креаторы и Кодеры» × Universal University × Школа 21.

---

## 1. Контекст и цели

### 1.1. Об институции
Центр «Зотов» — культурный центр в Москве (ул. Ходынская), расположенный в здании бывшего Хлебозавода №5 имени Зотова (1931 г., архитектор Г. Марсаков). Здание — памятник конструктивизма, имеет круглую (цилиндрическую) форму. Центр посвящён изучению русского авангарда и конструктивизма; в нём одновременно функционируют выставочные пространства, кинозалы, лекторий, книжный магазин, кафе, общественные зоны и пространства для мероприятий.

Сайты: centrezotov.ru, shop.centrezotov.ru

### 1.2. Проблема (из брифа)
Из-за архитектурных особенностей здания (круглая планировка, многофункциональность, кольцевые галереи) навигация остаётся сложной:
- высокий уровень дезориентации посетителей;
- сложные пользовательские маршруты;
- отсутствие интуитивной digital-навигации;
- высокая нагрузка на сотрудников ресепшн;
- отсутствие единой карты пространства.

### 1.3. Задача
Разработать концепцию и работающий прототип современной цифровой навигации, которая:
- упрощает ориентирование посетителя;
- снижает количество точек фрустрации;
- предлагает реалистичную и масштабируемую digital-концепцию;
- учитывает архитектурные особенности здания;
- создаёт современный пользовательский опыт.

### 1.4. Целевая аудитория
Посетители выставок и мероприятий 18–45 лет, новые посетители, гости с низким уровнем знакомства с пространством.

### 1.5. Пользовательские сценарии (jobs to be done)
1. **Найти конкретное мероприятие** внутри здания (по афише).
2. **Найти конкретное пространство** (лекторий, выставка, магазин, кафе).
3. **Быстро сориентироваться между этажами**.
4. **Построить маршрут** от текущей точки до цели.
5. **Спланировать день** — посетить несколько событий в оптимальном порядке.
6. **Навигация во время высокой нагрузки** (несколько событий одновременно).

### 1.6. Пространство использования
- сайт Центра (встраиваемый виджет);
- интерактивные стойки навигации внутри здания (киоск-режим);
- digital-экраны внутри центра;
- в перспективе — мобильное приложение.

## 2. Концепция «Зотов.Навигатор»

Веб-приложение с интерактивной 3D-картой здания. Ключевые принципы:
- **Навигация от события, а не от места** — пользователь выбирает мероприятие, система строит маршрут.
- **Адаптивные сценарии** — режимы «Я впервые», «На мероприятие», «Всё посмотреть», «С детьми».
- **Персонализированный маршрут дня** — планирование нескольких точек.
- **Стилистика конструктивизма** — минимализм, геометрия, палитра чёрный/белый/красный/серый, акценты на типографике.
- **Не перегружать пользователя** — простой UI, постепенное раскрытие информации.

## 3. Технический стек

| Слой | Технологии |
|---|---|
| Backend | Java 17, Spring Boot 3.x, Spring Web (REST), Spring Data JPA |
| База данных | H2 (in-memory, для хакатона) с возможностью переключения на PostgreSQL |
| Сборка | Maven |
| Frontend | Vanilla JS (ES modules) + Three.js (r160+) для 3D |
| Стилизация | Чистый CSS (custom properties), без фреймворков |
| Шаблонизатор | Опционально Thymeleaf для отдачи index.html, либо статические файлы из `src/main/resources/static/` |
| Документация API | springdoc-openapi (Swagger UI на `/swagger-ui.html`) |

**Почему такой стек:**
- Spring Boot — стандарт для Java backend, быстрый старт, всё из коробки.
- H2 в режиме in-memory — не нужно ничего ставить локально, данные грузятся из `data.sql` при старте.
- Three.js — индустриальный стандарт для 3D в браузере.
- Чистый JS — минимум зависимостей, проще встроить в WordPress (требование кейса) и масштабировать.

## 5. Модель данных

### 5.1. Сущности

**Floor (этаж)**
| Поле | Тип | Описание |
|---|---|---|
| id | Long | PK |
| number | int | номер этажа (1..5) |
| name | String | например, «Первый этаж — Форум» |
| heightMeters | double | высота этажа в метрах (для 3D) |
| radiusMeters | double | радиус цилиндра (для 3D) |
| color | String | hex-цвет для рендера |
| description | String | краткое описание |

**Zone (пространство/зона на этаже)**
| Поле | Тип | Описание |
|---|---|---|
| id | Long | PK |
| name | String | «Лекторий», «Кафе», «Выставочный зал A» |
| type | ZoneType (enum) | EXHIBITION, CINEMA, LECTURE, SHOP, CAFE, PUBLIC, ENTRANCE, RESTROOM, WARDROBE, ELEVATOR, STAIRS |
| floor | Floor | FK на этаж |
| angleStart | double | начальный угол сектора в градусах (0..360) — для круглого здания зоны удобно описывать сектором кольца |
| angleEnd | double | конечный угол сектора |
| radiusInner | double | внутренний радиус сектора |
| radiusOuter | double | внешний радиус сектора |
| color | String | hex-цвет |
| description | String | текст для попапа |
| iconKey | String | ключ иконки в UI |

**Event (мероприятие)**
| Поле | Тип | Описание |
|---|---|---|
| id | Long | PK |
| title | String | название |
| description | String | описание |
| startTime | LocalDateTime | начало |
| endTime | LocalDateTime | окончание |
| zone | Zone | FK на зону, где проходит |
| type | EventType | EXHIBITION, LECTURE, FILM, CONCERT, WORKSHOP, OTHER |
| posterUrl | String | URL афиши (опц.) |

### 5.2. Связи
- Floor 1 → N Zone
- Zone 1 → N Event
- Граф навигации: ребра между зонами вычисляются на лету сервисом RouteService по геометрии (соседние сектора одного этажа соединены, этажи связаны через зоны типа STAIRS/ELEVATOR).

### 5.3. Зачем сектора, а не координаты x/y
Здание круглое. Любую зону в нём естественно описать как сектор кольца: «от 30° до 90°, от 8 до 15 м от центра». Это:
- легко рендерится в Three.js (`RingGeometry` с `thetaStart`/`thetaLength`);
- совпадает с реальной планировкой Хлебозавода №5 (кольцевой конвейер);
- даёт визуально красивую конструктивистскую схему.

## 6. REST API

Базовый префикс: `/api`. Все ответы — JSON. CORS открыт для localhost при разработке.

### 6.1. Endpoints

| Метод | Путь | Описание |
|---|---|---|
| GET | `/api/floors` | Список всех этажей |
| GET | `/api/floors/{id}` | Этаж с зонами |
| GET | `/api/zones` | Все зоны (с фильтром `?type=CAFE&floorId=2`) |
| GET | `/api/zones/{id}` | Одна зона |
| GET | `/api/events` | Все события (фильтры `?date=2026-06-10&type=LECTURE`) |
| GET | `/api/events/upcoming` | Ближайшие события (от now) |
| GET | `/api/events/{id}` | Одно событие |
| POST | `/api/route` | Построить маршрут (см. ниже) |
| POST | `/api/route/multi` | Маршрут по нескольким точкам (план дня) |
| GET | `/api/search?q=...` | Глобальный поиск по зонам и событиям |

### 6.2. POST /api/route — формат

Запрос:
```json
{
  "fromZoneId": 1,
  "toZoneId": 12,
  "preferElevator": false
}
```

Ответ:
```json
{
  "totalDistanceMeters": 47.3,
  "estimatedSeconds": 75,
  "steps": [
    {
      "zoneId": 1,
      "floorNumber": 1,
      "instruction": "Старт от входа",
      "geometry": { "angleStart": 0, "angleEnd": 30, "radiusInner": 0, "radiusOuter": 5 }
    },
    {
      "zoneId": 5,
      "floorNumber": 1,
      "instruction": "Пройдите к лестнице",
      "geometry": { ... }
    },
    {
      "zoneId": 8,
      "floorNumber": 2,
      "instruction": "Поднимитесь на 2 этаж",
      "geometry": { ... }
    },
    {
      "zoneId": 12,
      "floorNumber": 2,
      "instruction": "Цель: Лекторий",
      "geometry": { ... }
    }
  ]
}
```

### 6.3. POST /api/route/multi — план дня
Запрос:
```json
{
  "fromZoneId": 1,
  "targetZoneIds": [12, 7, 20],
  "optimize": true
}
```
Ответ: упорядоченный маршрут через все точки (greedy ближайший сосед на хакатон достаточно).

## 7. 3D-модель здания (frontend, Three.js)

### 7.1. Базовая геометрия (упрощённая для прототипа)

Здание = 5 цилиндров одинакового радиуса, поставленных друг на друга. Каждый цилиндр — этаж. Один цилиндр строится через `THREE.CylinderGeometry(radius, radius, heightMeters, 64)` (64 сегмента, чтобы цилиндр был гладким).

Размеры (приблизительные, корректируются по реальному зданию):
- радиус — 20 м;
- высота этажа — 4 м;
- общая высота — 20 м.

### 7.2. Сцена
- `THREE.Scene` с тёмно-серым/чёрным фоном.
- Камера: `PerspectiveCamera`, fov 50°, позиция изометрически (например, (40, 30, 40), смотрит в (0, 10, 0)).
- Управление: `OrbitControls` — пользователь может крутить здание и приближать.
- Освещение: один `AmbientLight` + один `DirectionalLight` сверху-сбоку.

### 7.3. Иерархия объектов
```
scene
└── building (THREE.Group)
    ├── floor_1 (Group)
    │   ├── floor_1_shell    (CylinderGeometry — внешняя оболочка этажа)
    │   ├── zone_1_1         (Mesh из RingGeometry с thetaStart/thetaLength)
    │   ├── zone_1_2
    │   └── ...
    ├── floor_2 (Group)
    ├── floor_3 (Group)
    ├── floor_4 (Group)
    └── floor_5 (Group)
```
Каждая зона — отдельный `Mesh` с `userData = { zoneId, type, name }`. Это позволяет:
- ловить клики через `THREE.Raycaster`;
- управлять прозрачностью каждой зоны независимо;
- подсвечивать активную зону.

### 7.4. Режимы отображения
- **Базовый**: видны все 5 этажей, оболочки полупрозрачны (opacity 0.15), зоны внутри видны слабо.
- **Выбран этаж**: этажи выше скрываются (`visible = false`) или становятся полностью прозрачными; нужный этаж показывает зоны ярко.
- **Маршрут**: вдоль пути рисуется анимированная линия (`THREE.Line` через точки сегментов маршрута); камера плавно следует за маршрутом (lerp позиции).

### 7.5. Стилизация (конструктивизм)
- Палитра: чёрный `#0A0A0A`, белый `#F5F5F0`, красный `#E63329`, серый `#8C8C8C`.
- Цвета зон по типу: выставка — красный, кафе — жёлто-охра, лекторий — синий-стальной, кино — фиолетово-чёрный, магазин — белый, public — серый.
- Шрифт UI: гротеск (Inter / Manrope / Golos Text).
- Линии маршрута — толстые, контрастные (красный по чёрному).
- Анимации: плавные `TWEEN.js` или ручной lerp; transition при смене этажа — 400–600 мс.

### 7.6. Интерактив
- Клик по зоне → попап справа с описанием и кнопкой «Построить маршрут сюда».
- Hover на зону → подсветка + tooltip с названием.
- Переключатель этажей слева (вертикальный список 5 → 1).
- Поиск сверху: набор «Лекторий» → подсвечиваются все лектории на всех этажах.
- Кнопка «Афиша» открывает панель событий, клик по событию → маршрут к нему.

## 9. Сидинг данных (data.sql)

Тестовый набор для хакатона. Заполнить минимум:
- **5 этажей** (Floor) с реальными названиями зон Центра «Зотов»:
  - 1 этаж — «Форум» (общественное пространство, вход, гардероб, касса, магазин).
  - 2 этаж — выставочные пространства.
  - 3 этаж — лекторий, кинозал «Зотов.Кино».
  - 4 этаж — кафе, ещё одно выставочное пространство.
  - 5 этаж — пространство для мероприятий, терраса.
- **15–25 зон** (Zone), включая лестницы и лифт на каждом этаже.
- **6–10 событий** (Event) на ближайшую неделю — лекции, кинопоказы, выставки.

> Конкретное наполнение зонами должна согласовать команда с реальными данными от Центра — на хакатоне допустимы правдоподобные тестовые данные.

## 12. Алгоритм построения маршрута (RouteService)

Граф зон строится в памяти при старте приложения:
1. Для каждой зоны — вершина графа.
2. Рёбра внутри этажа: зоны, у которых пересекаются угловые границы или одна находится в радиусе другой (соседние сектора), соединены.
3. Рёбра между этажами: все зоны типа STAIRS/ELEVATOR на этаже N соединены с такими же на этаже N±1.
4. Вес ребра — приблизительное расстояние в метрах между центрами зон (с учётом разницы этажей).

Алгоритм поиска — **Dijkstra** (для хакатона можно даже BFS, граф маленький). Возвращается список вершин-зон, который затем превращается в `RouteStep[]` с инструкциями.

Для маршрута по нескольким точкам (`/route/multi`) — greedy nearest neighbour: из текущей точки идём в ближайшую невпосещённую, повторяем.

## 13. Соответствие критериям из брифа

| Критерий из кейса | Как закрыто |
